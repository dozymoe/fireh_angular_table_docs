FhFormIdMixin

Helps develop your own widget, provides consistent/uniform interface.

The reason this mixin were made was because of form cache (FhFormSessionStorage ).

Some forms are dynamically generated ( FhTableRow), so we need predictable dynamically generated id, and use this id to retrive and store form cache.

That doesn't stop this mixin from being used for other things.

The side effect is the complicated draft (form data being edited) related events that checks for form-id.

Let you use html attributes data-fhp-form-id and data-fhp-form-name with the widget.

If you use data-fhp-form-id the generated id is going to be exactly that.

If you use data-fhp-form-name the generated id is going to be that appended with form-data's identifier-fields, usually their 'id' property.

If you didn't supply form-data as argument it's going to be random, don't attach the form cache middleware.

Parameters

  1. scope

    Angular controller's $scope.

  2. attrs

    Angular controller's $attrs.

  3. item (optional)

    Form data, object bound to the form, a draft is a copy of this object with its fields modified by the form's input elements.
    Used to generate dynamic unique id.

  4. create (optional)

    Boolean, generate new form-id or reuse existing one in $scope.formId.

  5. name (optional)

    String, if not set will default to 'fh-form-'.

Results

  1. Return the generated unique id.
  2. Modify $scope, store unique id in $scope.formId.
